How data is structured in Supabase, how each service is wired to features, how errors are handled, and what gets tested.

Database Schema (Supabase / PostgreSQL)
Six tables. Users own Trips. Trips have Days. Days have Stops. Stops reference Places (cached from Google). Transport segments link stops.

Relationships
users→ 1:many →trips (user owns trips)
trips→ 1:many →days (trip spans days)
days→ 1:many →stops (day has ordered stops)
stops→ many:1 →places (stops reference cached place data)
stops→ 1:many →transport_segments (each stop has segments to next stop)
Services Wiring
Each external service is wrapped in a single file under src/services/. Features import via hooks only — never call services directly from components.
🛢️
Supabase
Auth + Database + Realtime
User sign-in (email + Google OAuth). All CRUD on trips, days, stops, places, transport_segments. Row-level security ensures users only access their own data. Realtime subscriptions for collaborative editing.
supabase.auth.signIn()
.from('trips').select()
.upsert()
.subscribe()
✦
OpenAI
AI Itinerary Generation + Suggestions
Called server-side via a Supabase Edge Function (never from client — keeps API key secret). Streams structured JSON (day → stops array) using GPT-4o with response_format: json_schema. "Suggest more" button calls the same edge function with existing stops as context.
supabase.functions.invoke('generate-trip')
supabase.functions.invoke('suggest-stops')
🗺️
Google Maps
Maps, Places, Directions
Map rendering via react-native-maps with Google Maps provider. Place search + autocomplete for adding stops. Place Details API populates the places table (name, category, thumbnail, hours, rating). Directions API fetches transport durations between stops and stores in transport_segments.

Firebase
Push Notifications (FCM)
Trip reminders sent via a Supabase scheduled Edge Function the morning of each travel day. Expo Notifications handles FCM token registration and foreground/background message display. Users can opt out per trip in Profile settings.

MMKV + AsyncStorage
Local Persistence & Offline Cache
MMKV stores auth tokens, theme preference, and last-known user prefs (fast synchronous reads). AsyncStorage holds JSON snapshots of trips for offline access — written whenever a trip is viewed online, read when offline. No SQLite needed at MVP scope.
MMKV.set('token', ...)
AsyncStorage.setItem('trip:id', JSON)
Error Handling
Errors are handled at the service boundary, never inside components. Users always see a calm, actionable message — never a raw error string.
Scenario Handling
OfflineNo network on launch Sage banner slides in. AsyncStorage snapshot loads trips. Network-only actions (AI, search) grey out with a tooltip. App is fully usable for viewing.
APIOpenAI generation fails Edge Function returns a structured error. App shows a calm inline message: "Couldn't generate — try again." No crash, no modal. Retry button in place of the shimmer.
APIGoogle Maps unavailable Transport segments show "–" instead of duration. Map hero shows a static placeholder tile. Place search shows "Search unavailable" in the input.
UserAuth token expired Supabase client auto-refreshes via its built-in token rotation. If refresh fails, user is quietly redirected to Sign In with a "Session expired" notice — no data loss.
UserRemark save fails Optimistic update shown immediately. On failure, value reverts silently and a subtle red underline appears on the field with a retry tap target.
OfflineMap app redirect — no app installed Linking.canOpenURL() checked before showing each map app in the picker. Only installed apps appear. Falls back to browser Google Maps if all deep-link apps are absent.
